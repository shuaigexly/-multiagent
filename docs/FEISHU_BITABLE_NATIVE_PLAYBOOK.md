# 飞书多维表格原生交付 Playbook

本文档只服务一个目标：让本仓库的多 Agent 产出尽可能回到飞书多维表格原生能力里消费，而不是依赖仓库外的自定义页面或人工拼装。

## 1. 官方能力对齐

基于飞书官方产品文档与开放平台入口，当前可以明确对齐的方向是：

1. OpenAPI 负责：
   - 建 Base / 表 / 字段 / 视图
   - 写记录与更新主表快照
   - 提供给多维表格自动化和视图消费的结构化字段

2. 多维表格原生负责：
   - 自动化 / 工作流
   - 看板 / 画册 / 甘特 / 表单 / 仪表盘
   - AI 总结、提醒、消息、任务、复核

3. 当前仓库应遵守的设计原则：
   - 所有“是否可汇报 / 是否要补数 / 是否要执行 / 是否要复核”的判断，优先写回主任务表
   - 前端页面只做驾驶舱，不应成为唯一的交付落点
   - 飞书 UI 内能配置的能力，不重复在代码里造另一套

## 2. 主表字段即工作流契约

`分析任务` 主表目前已经具备以下原生工作流消费字段：

- `最新评审动作`
- `最新评审摘要`
- `最新管理摘要`
- `汇报就绪度`
- `证据条数`
- `高置信证据数`
- `硬证据数`
- `待验证证据数`
- `进入CEO汇总证据数`
- `决策事项数`
- `需补数条数`
- `工作流路由`
- `工作流消息包`
- `工作流执行包`
- `待发送汇报`
- `待创建执行任务`
- `待安排复核`
- `建议复核时间`

这意味着：

1. 多维表格自动化无需理解七岗分析内部细节。
2. 它只需要读取这些字段，就能决定发消息、建任务、安排复核。

## 3. 推荐的多维表格原生自动化

建议在飞书多维表格 UI 中至少配置 3 条自动化：

1. 汇报自动化
   - 条件：`待发送汇报 = 是`
   - 动作：发送飞书消息 / 邮件
   - 正文：使用 `工作流消息包`

2. 执行动作自动化
   - 条件：`待创建执行任务 = 是`
   - 动作：创建飞书任务
   - 描述：使用 `工作流执行包`

3. 复核自动化
   - 条件：`待安排复核 = 是`
   - 时间：`建议复核时间`
   - 动作：提醒负责人 / 创建复核任务 / 发群消息

`交付动作` 表则用于承接这些自动化的回写或系统动作日志，至少应沉淀：

- `发送汇报`
- `创建执行任务`
- `创建复核任务`
- `自动跟进任务`
- `工作流记录`

`综合报告` 表已经可作为“高管一页纸”落点，关键字段：

- `重要风险`

- `高管一页纸`

建议后续把仪表盘重点挂在这张表，而不是只挂在前端页面。

## 6. 后续迭代优先级

如果继续强化“原生飞书交付闭环”，优先顺序如下：

1. 在 README 中单独拉出“多维表格原生自动化配置”章节。
2. 在 setup 引导记录中继续补充更细的自动化配置步骤截图或模板说明。
3. 如果未来开放平台开放“自动化规则创建”相关公开 API，再把这三条规则从“说明书”升级为“自动创建”。

## 7. 验收边界

当前 Playbook 里的原生能力对齐，分成两层：

1. 代码侧已经完成的结构与字段契约
2. 真实飞书权限下才可以完成的线上联调

当前仓库已能证明的是第 1 层：

- 主表字段契约存在
- 模板中心驱动任务写入和交付渲染
- 交付动作、复核历史、归档、自动化日志等结构已经进入代码主链路

当前仓库还不能单凭本地测试证明第 2 层：

- 飞书真实 Base 创建成功
- 飞书真实记录写入成功
- 飞书自动化真实生效
- 飞书任务 / 消息 / 提醒真实发送成功

因此本 Playbook 只能作为“可落地的原生设计说明”，不能替代真实飞书联调结论。
