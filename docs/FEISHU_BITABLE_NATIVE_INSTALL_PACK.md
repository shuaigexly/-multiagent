# 飞书多维表格原生安装包说明

本文档解释仓库里的 `native_manifest` 是干什么的，以及它如何继续推进“全部在多维表格原生”。

---

## 1. 为什么还需要安装包

当前仓库已经完成两类事情：

1. 代码侧已经把多 Agent 产出沉淀成可被飞书原生能力消费的字段契约。
2. setup 已经真实创建了多表、多视图、表单视图和模板中心。

但还有一类能力天然不该只停在 Python 或前端里：

- 自动化
- 工作流
- 仪表盘
- 高级权限 / 角色工作面

这些能力真正生效，必须继续落到飞书云侧原生对象中。

---

## 2. `native_manifest` 提供什么

`GET /api/v1/workflow/native-manifest` 现在会返回：

1. `install_order`
   - 明确先做什么、后做什么，避免 workflow / role / dashboard 顺序混乱

2. `command_packs`
   - 按 `advperm / form / workflow / dashboard / role` 输出 `lark-cli base` 命令模板
   - 作用是把“蓝图说明”进一步推进成“可执行的原生安装动作”

3. `markdown`
   - 一份适合直接复制到交接文档、项目 wiki、飞书文档里的原生安装说明

另外还支持：

- `POST /api/v1/workflow/native-manifest/apply`
- 或 setup 时传 `apply_native=true`

这会直接尝试把 `workflow / dashboard / role / form` 创建到飞书云侧，而不只是返回说明。
执行结果还会同步写入 `自动化日志` 表，触发来源为 `native_manifest.apply`。

---

## 3. 当前原生化边界

### 已原生创建

- Base 本体
- 主表与全部业务表
- 多数业务视图
- 任务收集表单视图

### 已原生蓝图化，但未云侧自动创建

- 自动化规则
- 工作流对象
- 仪表盘对象与图表块
- 高级权限角色

### 已给出落地包

- 安装顺序
- 命令模板
- 角色/工作流 JSON 骨架

这意味着仓库现在已经不只是“知道该做什么”，而是已经把后续原生安装动作编码成了系统产物。

---

## 4. 推荐落地顺序

1. 启用高级权限
2. 确认并共享任务收集表单
3. 创建路由工作流
4. 创建管理仪表盘和关键图表块
5. 创建高管 / 执行 / 复核角色工作面

---

## 5. 目标

最终目标不是让这个仓库永远替飞书执行所有动作，而是：

1. 代码负责生成分析、字段、决策信号和安装包。
2. 飞书多维表格原生对象负责执行真正的交付、分发、汇报、权限和可视化。
3. 前端只做驾驶舱和验收面，不替代飞书原生交付层。
