# 飞书多维表格交付闭环系统总蓝图

本文档不是零碎优化建议，而是当前仓库在“飞书多维表格原生交付闭环”方向上的完整目标态设计。

目标只有一个：

把这套系统从“七岗多 Agent 会分析”升级成“多维表格内可发起、可判断、可交付、可执行、可复核、可归档、可汇报”的业务系统。

---

## 1. 设计依据

本蓝图以飞书官方能力为边界，不自己幻想不存在的能力。

参考入口：

- OpenAPI Base overview：
  - https://open.feishu.cn/document/server-docs/docs/bitable-v1/bitable-overview
- 多维表格工作流：
  - https://www.feishu.cn/content/article/7595146589556788441
- 多维表格自动化：
  - https://www.feishu.cn/content/article/7581775868524104882
- 多维表格能力总览：
  - https://www.feishu.cn/content/base
- 多维表格新模块与高级权限：
  - https://www.feishu.cn/content/five-new-multidimensional-table-modules-feishu
- 多维表格 AI 能力与字段捷径：
  - https://www.feishu.cn/content/AsLfwRvl2iUT8lkDYuZcNvQenae

基于这些官方资料，当前可以明确接受的事实是：

1. OpenAPI 负责 Base / 表 / 字段 / 视图 / 记录。
2. 多维表格原生自动化和工作流适合负责触发、分支、循环、消息、任务、提醒、HTTP、AI 文本生成等流程动作。
3. 多维表格产品侧已经具备工作流、仪表盘、高级权限、AI 字段捷径、AI 节点等能力。
4. 所以本仓库的最优策略不是继续把所有东西做在前端页面里，而是把关键判断写回多维表格字段，再让原生工作流继续跑。

---

## 2. 北极星目标

完整目标态应满足以下 8 条：

1. 任务入口在飞书多维表格内完成。
2. 多 Agent 分析结果以结构化方式回写多维表格。
3. 系统能判断“是否能汇报、是否要拍板、是否要执行、是否要复核、是否要重跑”。
4. 这些判断能直接触发多维表格原生自动化与工作流。
5. 汇报动作、执行动作、复核动作必须可追溯。
6. 最终结论必须能沉淀为可归档、可验收、可复盘的交付结果。
7. 管理层、分析师、执行负责人、复核负责人看到的是不同视图，而不是同一堆字段。
8. 前端驾驶舱只是加速器，不是唯一落点。

## 3. Base 拓扑

建议把 Base 分成 3 类，而不是永远只有一个演示 Base：

### 3.1 模板 Base

用途：

- 固化标准表结构、标准字段、标准视图、标准自动化命名规范。

应包含：

- 所有正式字段
- 所有推荐视图
- 自动化命名约定
- 仪表盘模板
- 表单模板

### 3.2 生产 Base

用途：

- 真正承载某个业务团队、项目群、经营会的分析与交付。

原则：

- 不放演示数据
- 不放调试字段
- 只保留正式路由与正式仪表盘

### 3.3 验证 Base

用途：

- 测试新 schema、新自动化、新工作流、新仪表盘。

原则：

- 可承受脏数据
- 可频繁重置
- 用于新版本发布前验收

## 4. 目标态表结构

历史设计阶段曾从 8 张表扩展到 12 张表；当前仓库已经按 12 张表实现。

### 4.1 已实现的首批 8 表

1. `分析任务`
2. `岗位分析`
3. `综合报告`
4. `数字员工效能`
5. `📚 数据源库`
6. `证据链`
7. `产出评审`
8. `交付动作`

### 4.2 已补齐的 4 表

9. `复核历史`
10. `自动化日志`
11. `交付结果归档`
12. `模板配置中心`

### 4.3 每张表的职责

`分析任务`

- 任务 intake
- 状态机
- 路由判断
- 自动化触发字段
- 主交付快照

`岗位分析`

- 保留每个 Agent 的原始职责输出
- 支持跨角色对比
- 支持定位哪一岗导致结论弱

`综合报告`

- 面向管理层的整合结论
- 高管一页纸
- 汇报风险
- 决策事项

`数字员工效能`

- 统计 Agent 产能与活跃度
- 未来可扩展为 token / 时长 / 错误率 / 采纳率

`📚 数据源库`

- 统一维护数据资产
- 让任务通过 `引用数据集` 复用数据而不是重复粘贴

`证据链`

- 拆出 claim、evidence、confidence、usage
- 支撑“不是拍脑袋”的结论

`产出评审`

- 对结论做真实性、决策性、可执行性、闭环准备度评分
- 给出推荐动作

`交付动作`

- 记录汇报、执行、复核、自动跟进、工作流记录
- 形成交付审计时间线

`复核历史`

- 记录第几轮复核
- 记录为什么复核
- 记录复核前后结论差异

`自动化日志`

- 记录具体是哪条自动化、哪个工作流节点、何时触发、是否成功

`交付结果归档`

- 记录最终对谁汇报、谁采用、谁执行、截止时间、完成结果

`模板配置中心`

- 管理消息模板、执行模板、复核模板、审批模板、字段映射模板

## 6. 目标态状态机与路由

### 6.1 状态机

`待分析 -> 分析中 -> 已完成 -> 已归档`

不建议把太多业务逻辑堆进 `状态` 单字段里。

真正的业务分流应该放在 `工作流路由`。

### 6.2 标准路由

- `直接汇报`
- `等待拍板`
- `直接执行`
- `补数复核`
- `重新分析`

### 6.3 路由后的业务动作

`直接汇报`

- 发消息
- 发邮件
- 进入高管视图
- 等待采用反馈

`等待拍板`

- 发高管卡片
- 触发审批或确认动作
- 等待拍板结果回写

`直接执行`

- 创建飞书任务
- 发送给执行 owner
- 设置执行截止时间

`补数复核`

- 创建复核任务
- 设置复核 owner
- 到达复核时间提醒

`重新分析`

- 生成重跑记录
- 保留旧版本
- 标记重跑原因

## 8. 自动化与工作流完整设计

根据飞书官方资料，自动化适合做简单触发动作，工作流适合做复杂分支、循环和 AI 节点。

因此目标态应拆成两层：

### 8.1 自动化层

处理单步、稳定、低分支动作：

1. 新增任务提醒
2. 完成后自动发汇报
3. 待执行时自动建任务
4. 待复核时自动提醒
5. 超时未执行提醒
6. 超时未复核提醒
7. 交付结果归档提醒
8. 已完成任务自动归档

### 8.2 工作流层

处理有分支、有循环、有 AI 节点的复杂交付：

1. 根据 `工作流路由` 决定走哪条分支
2. 根据 `汇报对象` 选择不同消息模板
3. 根据 `输出目的` 选择经营诊断 / 管理决策 / 执行跟进模板
4. 失败动作进入补救分支
5. 多执行 owner 进入循环创建任务
6. 高风险任务进入审批分支

### 8.3 AI 工作流修改路径

这是非常关键的一层，也是这套系统接下来最值得利用的飞书原生能力：

1. 用 AI 对话式创建或修改工作流
2. 用 AI 生成文本节点生成个性化汇报消息
3. 用 AI 分类节点给任务自动打标
4. 用 AI 总结节点生成短版管理摘要
5. 用 AI 为仪表盘生成解释文本

注意边界：

- 根据官方资料，工作流支持 `AI 生成文本`，但不支持 `AI 更新记录` 这类工作流执行操作。
- 因此推荐做法不是指望工作流直接“自己改所有字段”，而是：
  - 代码先把结构化字段写好
  - 工作流用 AI 节点补生成消息、摘要、解释、分类文本

## 9. 仪表盘目标设计

飞书官方能力已经支持多表仪表盘，因此不应只靠前端做驾驶舱。

### 9.1 高管仪表盘

核心问题：

- 哪些任务可以直接汇报？
- 哪些任务需要拍板？
- 当前最大的风险是什么？

核心指标：

- 已完成任务数
- 可直接汇报任务数
- 等待拍板任务数
- 高就绪度任务数
- 风险任务数

### 9.2 交付运营仪表盘

核心问题：

- 任务有没有真的流转到交付动作？
- 哪些动作失败了？

核心指标：

- 交付动作总数
- 成功动作数
- 失败动作数
- 汇报动作完成率
- 自动跟进转化率

### 9.3 证据质量仪表盘

核心问题：

- 这套系统输出为什么强或弱？

核心指标：

- 硬证据数
- 待验证证据数
- 高置信证据占比
- 进入 CEO 汇总证据占比

### 9.4 复核中心仪表盘

核心问题：

- 哪些结论还不可靠？

核心指标：

- 待复核任务数
- 建议重跑数
- 复核逾期数
- 平均复核闭环时长

### 9.5 数据资产健康仪表盘

核心问题：

- 任务为什么会补数？

核心指标：

- 数据集数量
- 高可信数据集占比
- 最近校验过期数据集数
- 被高频引用的数据集 Top N

## 10. 产出提升的核心方法

如果目标是提升系统产出，关键不在“接更多 Agent 框架”，而在下面这 12 件事。

### 10.1 强化任务输入

必须让任务不只是标题，而是带：

- `目标对象`
- `输出目的`
- `成功标准`
- `约束条件`
- `业务阶段`
- `引用数据集`

### 10.2 把数据资产独立出来

让任务引用数据集，而不是每次重复贴一大段数据源。

### 10.3 把证据从长文本里剥离

用 `证据链` 表去沉淀 claim / source / confidence / usage / grade。

### 10.4 加一道自动评审

没有评审，系统只会“会说”；有评审，系统才知道“能不能交付”。

### 10.5 让 CEO 输出可路由

不是所有结论都直接发老板。

必须压缩成明确路由：

- 汇报
- 拍板
- 执行
- 复核
- 重跑

### 10.6 把消息和执行包模板化

不要让每次交付都从原始长文临时拼。

### 10.7 对失败动作做补救分支

只记录失败不够，要有补救设计：

- 重试
- 改走邮件
- 改走人工接管
- 改走复核

### 10.8 对重跑保留版本链

否则结论一变就无法复盘。

### 10.9 对复核建立 SLA

否则 `待安排复核` 只是一个漂亮字段。

### 10.10 对执行建立 owner 与截止时间

否则 `待创建执行任务` 只会停留在“建议”。

### 10.11 用 AI 节点做“二次表达”，不要替代结构化判断

AI 节点适合：

- 改写为高管语言
- 生成群消息
- 生成邮件主题
- 生成仪表盘解释

不适合替代核心路由判断。

### 10.12 建立结果归档

最终应沉淀：

- 发给谁
- 谁采纳
- 谁执行
- 是否完成
- 复盘结论是什么

## 12. 表单入口设计

官方资料显示自动化与表单天然适合组合，因此目标态不能只让用户进入表里手工填记录。

建议至少有 4 套表单：

1. `经营诊断任务表单`
2. `管理决策任务表单`
3. `执行跟进任务表单`
4. `补数复核任务表单`

每套表单都应把字段顺序重排成业务语言，而不是数据库语言。

---

## 13. 模板体系

系统想稳定产出，必须模板化。

模板至少分为 5 类：

1. 汇报消息模板
2. 高管卡片模板
3. 执行任务模板
4. 复核任务模板
5. 归档摘要模板

模板中心至少应支持这些维度：

- 适用路由
- 适用输出目的
- 适用业务阶段
- 适用目标对象
- 使用字段
- 版本号

---

## 14. 目标态验收标准

如果以后说“这套系统做完整了”，至少要满足下面这些验收条件：

1. 用户可通过表单或主表新增任务。
2. 任务可以引用数据源库中的数据集。
3. 七岗输出能写回岗位分析。
4. CEO 结论能写回综合报告。
5. 证据链可独立查看和分级。
6. 自动评审可给出推荐动作。
7. 主表可形成明确工作流路由。
8. 多维表格自动化能基于这些字段触发。
9. 交付动作表能回放完整动作链。
10. 复核历史能记录前后差异。
11. 交付结果归档能承接最终采用与执行状态。
12. 飞书仪表盘能替代大部分临时汇报页面。

- 配置完整原生自动化与工作流

- 做 AI 节点参与的消息改写、分类、仪表盘解释

### Phase E

- 做多 Base 模板化复制
- 做版本升级与 schema migration

## 16. 结论

这套系统真正的天花板，不是“还能接多少 Agent 框架”，而是：

1. 多维表格是不是成为真实业务系统。
2. 分析结果是不是能变成可执行动作。
3. 执行动作是不是能被追溯。
4. 复核与归档是不是能形成闭环。
