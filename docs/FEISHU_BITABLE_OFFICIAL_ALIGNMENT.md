# 飞书多维表格官方文档对齐改造说明

本文档用于把本仓库的多 Agent 多维表格工作流，与飞书官方多维表格文档分支进行一一对齐。

核心原则：

1. 以飞书官方多维表格文档为准。
2. 先承认开放平台与产品 UI 的边界，再决定该由代码做什么、该由多维表格原生能力做什么。
3. 所有多 Agent 产出，优先沉淀到多维表格字段与视图中，再由多维表格原生自动化、工作流、仪表盘、AI 功能消费。

## 2. 官方分支结构与仓库对齐

根据开放平台该分支目录，当前与仓库最相关的能力可以按以下模块理解：

### 2.1 多维表格 app

官方含义：

- 创建 Base
- 访问 Base 资源
- 进行应用级权限控制

仓库对齐：

- 已实现 `setup_workflow()` 创建 Base
- 已实现初始化表、字段、视图、种子记录
- 已实现初始权限授权与共享链接逻辑

目标：

- Base 不是“临时存储”，而是多 Agent 交付闭环的最终落点

### 2.2 数据表 table

官方含义：

- 创建 / 管理表
- 组织多表结构

仓库对齐：

- 已拆成 `分析任务 / 岗位分析 / 综合报告 / 数字员工效能 / 数据源库 / 证据链 / 产出评审`

目标：

- 主表负责流程调度与交付快照
- 输出表负责深度分析内容
- 证据 / 评审表负责可信度与复核闭环

### 2.3 视图 view

官方含义：

- 以不同方式展示同一份数据
- 通过筛选、分组、看板、甘特、画册等方式表达业务状态

仓库对齐：

- 已通过 OpenAPI 自动创建多个 grid / kanban / gallery / gantt / form 视图
- 已通过公开支持的 `filter_info` 做筛选视图
- 对于分组字段、画册封面字段等当前无法通过公开 API 完整配置的内容，保留 UI 引导记录

目标：

- 让飞书内看板本身成为管理者汇报入口，而不是只有仓库前端能看

### 2.4 记录 record

官方含义：

- 记录是多维表格的最核心业务实体
- OpenAPI 负责增删改查、批量更新、检索

仓库对齐：

- 已实现 list / create / update / delete / search / batch_update / batch_delete
- 已使用记录驱动七岗多 Agent 调度
- 已把评审、证据、CEO 结果回写为主表交付快照

目标：

- 每条任务记录最终都要能独立承载“能否汇报 / 是否执行 / 是否复核”的完整语义

### 2.5 字段 field

官方含义：

- 字段是业务结构设计的核心
- 字段类型决定系统是否能被原生自动化、仪表盘、公式和 AI 能力消费

仓库对齐：

- 已广泛使用 SingleSelect / Number / Rating / Progress / CreatedTime / ModifiedTime / DateTime / Attachment / Checkbox 等字段
- 已新增主表交付快照字段
- 已新增工作流路由字段
- 已新增证据等级字段
- 已新增高管一页纸字段

目标：

- 把多 Agent 结果压成“原生工作流可消费字段”，而不只是长文本

### 2.6 字段分组 field group

官方含义：

- 字段组织与显示分层
- 适合复杂业务系统的配置体验优化

仓库现状：

- 尚未基于公开 API 明确实现字段分组管理

改进方向：

- 后续若该能力在当前应用权限和 API 范围内可稳定使用，可将主表字段按：
  - 任务输入
  - 调度状态
  - 交付快照
  - 原生工作流交接包
  进行分组

### 2.7 仪表盘 block

官方含义：

- 多表聚合分析
- 图表、统计、管理视图

仓库现状：

- 当前前端实现了汇报驾驶舱
- 多维表格内侧仍主要依赖视图层，还没有形成一份“仪表盘规划说明”

改进方向：

- 后续应把 `综合报告`、`分析任务`、`证据链` 的关键字段设计成适合仪表盘消费的统计口径：
  - 直接采用数
  - 待补数数
  - 硬证据数
  - 待验证证据数
  - 路由分布
  - 复核积压数

### 2.8 表单 form

官方含义：

- 面向业务方或外部协作者的数据采集入口

仓库对齐：

- 已自动创建 `📥 需求收集表` 视图
- 已引导用户把表单作为任务入口使用

目标：

- 让任务输入标准化，减少 prompt 漂移

### 2.9 高级权限

官方含义：

- 面向更细粒度的数据访问控制

仓库现状：

- 已做基础成员授权与分享
- 尚未做更细的记录级 / 视图级权限协同设计说明

改进方向：

- 后续可根据角色把：
  - 高管只看 `综合报告`
  - 分析师看 `证据链 / 岗位分析`
  - 业务执行人看 `分析任务`
  的权限模型写成明确方案

### 2.10 自动化流程 workflow

官方含义：

- 多维表格原生触发器与动作
- 可联动消息、任务、审批、提醒、AI 节点等

仓库对齐：

- 已把主表设计成可被自动化直接消费：
  - `工作流路由`
  - `工作流消息包`
  - `工作流执行包`
  - `待发送汇报`
  - `待创建执行任务`
  - `待安排复核`
  - `建议复核时间`

目标：

- 让代码负责“生成决策信号”，让多维表格原生工作流负责“执行动作”

## 3. 已经落地的关键改造

这轮已经落地到代码中的关键改造：

1. 主表交付快照字段
   - 支持直接在 `分析任务` 表里判断汇报就绪度、证据质量和后续路由

2. 原生工作流交接包
   - 自动生成 `工作流消息包 / 工作流执行包`
   - 适合在飞书自动化中直接引用

3. 工作流路由
   - `直接汇报 / 等待拍板 / 直接执行 / 补数复核 / 重新分析`

4. 证据等级
   - `硬证据 / 推断 / 待验证`

5. 高管一页纸
   - `一句话结论 / 首要动作 / 汇报风险 / 高管一页纸`

6. 原生视图增强
   - `📣 待发送汇报`
   - `🧾 待创建执行任务`
   - `🗓 待安排复核`
   - `🧭 工作流路由`
   - `🧱 硬证据`
   - `🟡 待验证`

7. 交付动作表
   - 已新增 `交付动作` 表
   - 已回写 `发送汇报 / 创建执行任务 / 创建复核任务 / 自动跟进任务 / 工作流记录`
   - 目标是让飞书多维表格本身能回放整条交付链，而不是只看最终结论

## 4. 建议继续实现的内容

以下改造建议均以飞书官方多维表格能力为方向，不再单独造轮子：

### P0

1. 在 README 中增加“多维表格原生自动化配置”章节
2. 给 3 条推荐自动化规则补成可复制的 UI 配置步骤
3. 给 `综合报告` 和 `分析任务` 设计多维表格仪表盘指标说明

### P1

1. 研究字段分组在当前 API 能力下是否可稳定接入
2. 把高级权限模型写成角色化方案
3. 为 `表单` 入口设计更清晰的任务 intake 模板

### P2

1. 若飞书开放平台未来公开“自动化规则创建”API，则把当前“说明书式自动化”升级成自动初始化
2. 将多维表格 AI 能力与主表字段进一步对齐，例如：
   - AI 总结字段
   - AI 分类字段
   - AI 推荐关联记录

---

## 5. 结论

本仓库后续所有与飞书多维表格有关的改造，都应遵循以下目标：

1. 多 Agent 负责分析。
2. 多维表格负责沉淀。
3. 原生自动化负责执行。
4. 仪表盘负责汇报。

这样系统才真正是“面向飞书交付闭环的多 Agent 多维表格应用”，而不是“借用了多维表格做存储的独立系统”。
