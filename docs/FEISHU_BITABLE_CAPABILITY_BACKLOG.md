# 飞书多维表格能力缺口与改造 Backlog

本文档是 `FEISHU_BITABLE_OFFICIAL_ALIGNMENT.md` 的执行版。

它不再只讲原则，而是明确记录：

1. 官方多维表格能力分支下，当前仓库还缺什么。
2. 为什么缺。
3. 应该怎么改。
4. 优先级是什么。
5. 改完后对“飞书交付闭环多 Agent 应用”有什么价值。

---

## 0. 总目标

最终目标不是“仓库能调通多维表格 API”，而是：

1. 用户在飞书多维表格里发起任务。
2. 七岗多 Agent 在后台完成分析。
3. 结果沉淀为主表快照、证据链、评审、综合报告。
4. 多维表格原生自动化继续完成：
   - 发汇报
   - 建执行任务
   - 安排复核
   - 做仪表盘
   - 做表单入口
5. 前端页面只是驾驶舱，不是唯一交付入口。

---

## 0.1 当前状态口径

从 `v8.6.20-r6` 开始，仓库已经不再把所有原生能力都混成一句“未来可做”，而是明确拆成 3 类：

1. `created`
   - setup 后已经真实创建在 Base 里。
   - 目前典型代表：主表、多业务表、原生视图、任务收集表单视图。

2. `manual_finish_required`
   - 基础对象已经有了，但最后一步必须在飞书 UI 里补完。
   - 目前典型代表：表单共享链接、部分视图分组/封面配置。

3. `blueprint_ready`
   - 仓库已经把字段契约、角色工作面、指标口径和操作步骤定义好，但飞书云侧对象还没有自动创建。
   - 目前典型代表：自动化、工作流、仪表盘、角色权限。

这意味着 backlog 的重点已经从“有没有设计”转成“哪些原生对象还没有完全落地到飞书里”。

## 1. App 层

### 已实现

- 创建 Base
- 初始权限授权
- 分享链接
- 初始化多表、多视图、种子数据

### 缺口

- 缺少“多 Base 管理策略”
- 缺少“环境隔离策略”说明
- 缺少“模板 Base / 业务 Base / 验证 Base”分层方案

### 改进目标

- 把 Base 从“一个 demo 实例”升级成“可多环境、可模板化、可演进的业务交付载体”

### 建议改造

1. 增加 `Base 类型` 约定
   - 模板 Base
   - 生产 Base
   - 验证 Base

2. 增加 `setup` 模式
   - `--template-only`
   - `--seed-demo`
   - `--prod-empty`

3. 记录 Base 元数据
   - 版本号
   - schema 版本
   - 初始化时间
   - 来源模板

### 价值

- 方便在飞书内复制和迭代不同业务场景

## 2. Table 层

### 已实现

- 主表、输出表、报告表、效能表、数据源表、证据链、评审表、交付动作表

### 缺口

- 缺少“复核历史表”
- 缺少“自动化执行日志表”
- 缺少“交付结果归档表”

### 改进目标

- 让多维表格不只保存分析结果，还保存原生工作流执行历史

### 建议改造

1. 新增 `复核历史` 表
   - 原任务
   - 复核轮次
   - 复核原因
   - 复核结果
   - 新旧结论差异

2. 新增 `自动化日志` 表
   - 哪条主表记录触发了什么飞书动作
   - 是否成功

3. 新增 `交付结果归档` 表
   - 汇报对象
   - 汇报版本
   - 最终采用结论
   - 执行 owner
   - 截止时间
   - 完成状态

### 价值

- 真正形成飞书交付闭环，而不是只完成分析闭环
- 把“任务完成”升级成“交付动作可追溯 + 结果归档可验收”

### 已新增的 `交付动作` 表能力

- 系统当前可沉淀 `发送汇报 / 创建执行任务 / 创建复核任务 / 自动跟进任务 / 工作流记录`
- 可直接在飞书里追查某条任务到底有没有真正进入“汇报、执行、复核”
- 已适合作为多维表格自动化和管理复盘的动作审计底座

## 3. View 层

### 已实现

- grid / kanban / gallery / gantt / form 视图
- 多个筛选视图
- 工作流路由视图

### 缺口

- 视图命名体系还不够统一
- 缺少“角色视角视图”
- 缺少“高管视角视图”
- 缺少“执行视角视图”
- 缺少“复核中心视图”

### 改进目标

- 让不同角色直接在飞书里看到属于自己的工作面板

### 建议改造

1. 高管视图
   - 只看 `已完成`
   - 只看 `直接汇报 / 等待拍板`
   - 强调 `一句话结论 / 汇报风险 / 汇报就绪度`

2. 执行视图
   - 只看 `待创建执行任务 = 是`
   - 强调 `工作流执行包`

3. 复核视图
   - 只看 `待安排复核 = 是`

4. 动作运营视图
   - 只看 `交付动作` 表
   - 按 `动作状态 / 动作类型 / 工作流路由` 切片
   - 用于识别哪些交付动作失败、被跳过、重复触发
   - 按 `建议复核时间` 排序

4. 证据视图
   - `硬证据`
   - `待验证`
   - `进入CEO汇总`

5. 数据资产视图
   - 按 `可信等级`
   - 按 `适用任务类型`

### 价值

- 降低飞书内人工筛选成本

## 4. Record 层

### 已实现

- create / list / get / update / delete
- search / batch_update / batch_delete

### 缺口

- 缺少“记录级 diff / 变更追踪”能力
- 缺少“重跑前后版本对比”
- 缺少“归档策略”

### 改进目标

- 让每条任务不仅有当前状态，还有演进历史

### 建议改造

1. 任务重跑时保留版本号
2. 复核任务与原任务建立更明确的版本链
3. 对已完成但过期的任务增加自动归档策略

### 价值

- 方便复盘“为什么这次结论变了”

## 5. Field 层

### 已实现

- 任务状态字段
- 交付快照字段
- 工作流路由字段
- 证据等级字段
- 高管一页纸字段

### 缺口

- 缺少“角色字段”
- 缺少“业务归属字段”
- 缺少“汇报对象字段标准化”
- 缺少“任务来源字段”
- 缺少“自动化执行状态字段”

### 改进目标

- 把字段做成真正的业务模型，而不只是技术模型

### 建议改造

1. 新增 `任务来源`
   - 手工创建
   - 表单提交
   - 跟进任务
   - 复核任务
   - 外部系统同步

2. 新增 `业务归属`
   - 产品
   - 增长
   - 内容
   - 运营
   - 财务

3. 新增 `汇报对象级别`
   - 负责人
   - 部门管理层
   - CEO / CXO

4. 新增 `自动化执行状态`
   - 未触发
   - 执行中
   - 已完成
   - 失败

### 价值

- 让多维表格原生自动化条件更稳定、更业务化

## 10. Workflow / 自动化流程

### 已实现

- 主表原生自动化消费字段
- 交付路由
- 汇报/执行/复核三条主路线

### 缺口

- 自动化规则还只是“建议配置”，没有形成可复制操作手册
- 缺少失败重试与动作日志机制

### 改进目标

- 让飞书多维表格原生工作流成为系统后半段执行引擎

### 建议改造

1. 写成三条标准自动化模板
   - 汇报模板
   - 执行模板
   - 复核模板

2. 每条模板写清：
   - 触发条件
   - 使用字段
   - 输出动作
   - 失败补救

3. 继续扩展 `交付动作` 表，把更多自动化节点回写进去
4. 给 `交付动作` 表增加责任人、执行渠道、执行批次
5. 为 `交付动作` 表补充“失败原因归类”字段

### 价值

- 从“分析工具”升级为“交付系统”

## 13. 结论

本仓库还远没改完。

真正要做到“面向飞书交付闭环的多 Agent 多维表格应用”，后续必须继续把以下几件事做透：

1. 多维表格主表要成为工作流操作台。
2. 多维表格原生自动化要成为执行引擎。
3. 多维表格仪表盘要成为汇报界面。
4. 前端页面只做增强，不做唯一落点。

这份 backlog 就是后续所有改造的执行基线。
