# 飞书多维表格自动化与工作流模板库

本文档把这套系统应该配置的自动化和工作流一次性列全，避免后续还靠口头描述。

补充口径：

- 本文档中的 `A* / W*` 已经同步到代码返回的 `native_assets` 蓝图里。
- 额外扩展了一条 `A2b 待拍板汇报提醒`，用于把 `待发送汇报` 的 `等待拍板` 分支从 `直接汇报` 中拆开。
- 当前状态是“蓝图就绪”，不是“飞书云侧对象已自动创建”。
- 验收时请结合 `native_assets.manual_finish_checklist`，在飞书 UI 里逐条补齐。

设计原则：

1. 简单单步动作优先用自动化。
2. 有条件分支、循环、补救逻辑的优先用工作流。
3. 所有流程都尽量回写 `交付动作`、`自动化日志`、`交付结果归档`。

## 1. 自动化模板

### A1 新任务入场提醒

触发：

- `添加记录时`

条件：

- `状态 = 待分析`

动作：

- 发送飞书消息给分析运营群
- 在 `交付动作` 表写一条 `工作流记录`

目的：

- 让新任务进入可见范围

### A2 直接汇报自动提醒

触发：

- `新增/修改记录满足条件时`

条件：

- `待发送汇报 = 是`
- `工作流路由 = 直接汇报`

动作：

- 发送飞书消息或邮件
- 正文引用 `工作流消息包`
- 回写 `自动化日志`
- 具体汇报动作由 `W2 汇报分支工作流` 创建，避免重复动作

### A3 执行任务自动创建

触发：

- `新增/修改记录满足条件时`

条件：

- `待创建执行任务 = 是`

动作：

- 发送执行提醒
- 回写 `自动化日志`
- 具体执行动作由 `W4 执行分支工作流` 创建，避免重复执行单

### A4 复核提醒

触发：

- `到达记录中的时间时`

条件：

- `待安排复核 = 是`

动作：

- 发送提醒
- 回写 `自动化日志`
- 具体复核动作由 `W5 复核分支工作流` 创建，避免重复复核单

### A5 异常升级提醒

触发：

- `新增/修改记录满足条件时`

条件：

- `异常状态 = 已异常`

动作：

- 发送异常升级提醒
- 保留当前任务实际路由，不伪造 `重新分析`
- 把人工接管入口写进 `自动化日志`

### A6 超时未复核提醒

触发：

- `到达记录中的时间时`

条件：

- `待安排复核 = 是`
- `建议复核时间` 已超时

动作：

- 查找 `建议复核时间 < 当前时间` 且 `待安排复核 = 是` 的记录
- 提醒复核 owner

### A7 失败动作报警

触发：

- `新增/修改记录满足条件时`

条件：

- 在 `交付动作` 表里 `动作状态 = 执行失败`

动作：

- 发报警消息
- 写入失败告警日志

### A8 归档提醒

触发：

- `新增/修改记录满足条件时`

条件：

- `状态 = 已完成`
- `是否已执行落地 = 是`

动作：

- 切换到复盘负责人责任面
- 创建归档跟进动作
- 通知补齐 `交付结果归档`

## 2. 工作流模板

### W1 路由总分发工作流

触发：

- `状态 = 已完成`

逻辑：

- If `工作流路由 = 直接汇报` -> 汇报分支
- If `工作流路由 = 等待拍板` -> 拍板分支
- If `工作流路由 = 直接执行` -> 执行分支
- If `工作流路由 = 补数复核` -> 复核分支
- If `工作流路由 = 重新分析` -> 重跑分支

### W2 汇报分支工作流

触发：

- `待发送汇报 = 是`
- `工作流路由 = 直接汇报`

动作：

- 切换到汇报对象责任面
- 发送管理汇报提醒
- 写 `直接汇报` 动作到 `交付动作`

### W3 拍板分支工作流

触发：

- `待拍板确认 = 是`

动作：

- 切换到拍板人责任面
- 发送拍板提醒
- 写 `等待拍板` 动作到 `交付动作`

### W4 执行分支工作流

动作：

- 切换到执行人责任面
- 写 `直接执行` 动作到 `交付动作`
- 延迟后再次提醒执行进展

### W5 复核分支工作流

动作：

- 切换到复核人责任面
- 写复核动作到 `交付动作`
- 延迟后再次提醒复核进展

### W6 重跑分支工作流

动作：

- 保留重跑原因和结论差异
- 在 `复核历史` 写入重跑记录
- 提醒复核负责人继续安排重跑

### W7 动作失败补救工作流

触发：

- `交付动作.动作状态 = 执行失败`

动作：

- 发送失败补救提醒
- 写补救日志
- 推动人工接管

### W8 群消息驱动工作流

触发：

- `自动化日志.触发来源 = im.command`

用途：

- 由群命令解析器先把消息写入 `自动化日志`
- 再由工作流生成后续催办 / 复核 / 复盘跟进动作

### W9 仪表盘自动推送工作流

触发：

- `状态 = 已归档`

动作：

- 提示整理仪表盘截图
- 附带管理摘要和推送日志

### W10 数据资产校验工作流

触发：

- `待安排复核 = 是`

动作：

- 检查 `引用数据集 / 数据源 / 需补数事项`
- 通知复核负责人先做数据资产校验再推进复核

## 3. AI 节点推荐用法

根据官方资料，当前最适合接入本系统的 AI 能力有：

1. `AI 生成文本`
   - 生成高管口径摘要
   - 生成群消息卡片正文
   - 生成邮件主题和正文

2. `AI 分类`
   - 给任务自动打风险标签
   - 给任务自动打业务主题标签

3. `字段捷径`
   - 生成更短的展示摘要
   - 对文本做分类与总结

4. `仪表盘智能解释`
   - 自动解释图表
   - 随仪表盘推送附带洞察

## 4. 推荐命名规范

自动化：

- `AUTO_任务入场提醒`
- `AUTO_直接汇报自动提醒`
- `AUTO_待拍板汇报提醒`
- `AUTO_执行任务自动创建`
- `AUTO_复核提醒`

工作流：

- `WF_路由总分发`
- `WF_直接汇报`
- `WF_等待拍板`
- `WF_直接执行`
- `WF_补数复核`
- `WF_重新分析`

---

## 5. 最终要求

所有自动化和工作流都应满足三条要求：

1. 能看懂是给谁服务的。
2. 能看懂失败后怎么补救。
3. 能回写到 `交付动作` 或 `自动化日志`，避免“看起来自动化很多，但没人知道它到底跑没跑”。
