# 议题树分析法（Issue Tree）

## 核心原则：假设驱动，不是数据驱动

不要先收集所有数据再下结论——先提出最可能的假设，再用数据验证。

## 四步构建流程

### Step 1：精确定义核心问题（一句话）
❌ 差：分析一下业务  
✅ 好：为什么本季度 DAU 同比下降 23%？

### Step 2：提出 2-3 个互斥假设
- 假设 A：获客减少（新用户进来的少了）
- 假设 B：留存变差（老用户流失加速）
- 假设 C：统计口径变更（数字变化不是真实变化）

### Step 3：MECE 拆解验证路径
```
DAU 下降 23%
├── 新用户减少？
│   ├── 付费渠道 ROI 下降？（查广告成本/注册量）
│   └── 自然流量萎缩？（查 SEO / 口碑传播）
├── 老用户流失？
│   ├── Day1 留存跌了？（Onboarding 问题）
│   ├── Day7 留存跌了？（核心功能价值问题）
│   └── 哪个用户群流失最严重？（分层分析）
└── 统计口径变更？
    └── 查近期埋点/数据口径变更记录
```

### Step 4：按影响面优先验证
影响最大的假设先验证，快速排除噪音

## 质量检查清单
- [ ] 每层节点互斥（不重叠）且穷举（不遗漏）
- [ ] 叶节点可以用具体数据直接回答
- [ ] 树深度 ≤ 4 层（更深说明问题定义模糊）
- [ ] 每个分支能估算对母问题的贡献比例

## 结论标注规范
每个结论必须标注置信度：
- **[确定]**：有直接数据支撑
- **[推断]**：基于间接证据的逻辑推导
- **[假设]**：缺乏数据，基于经验的合理猜测

## 时间紧迫时的快速版
用"3×3 结构"：核心问题 → 3 个假设 → 每假设 3 个验证点
9 个验证点通常覆盖 80% 的解释空间，30 分钟可完成初步分析
